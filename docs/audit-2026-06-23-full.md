## 🔴 HIGH 高危 (7)

### [auth-session-2] No rate-limit / lockout on /auth/login — unthrottled password brute-force
- **等级**: HIGH  ·  **类别**: security  ·  **置信**: high
- **位置**: `internal/api/router.go:21`, `internal/api/auth_handlers.go:23-50`
- **现象/根因**: POST /auth/login is wired to a bare loginHandler with no limiter, no per-IP/per-username failure counter, and no lockout (router.go:21). Each attempt does one scrypt (N=32768) verify (or DummyVerify on unknown user), which adds CPU cost but is not a defense against online guessing of weak passwords — an attacker can run unlimited attempts. The codebase has NO rate-limit/lockout/attempt-tracking primitives anywhere (grep for ratelimit/lockout/attempt/throttle/limiter found only dispatch-failover 'attempts', unrelated). Note: a sibling Node product (Nexaxis 中控台) reportedly has login throttling/brute-force lockout, but this Go Tower rewrite does not — the protection was not carried over.
- **线上证据**: Static: only one /auth/login route registered, no middleware. Did not exercise the live login endpoint (would be a write/credential action).
- **修复建议**: Add per-username + per-IP failure counters with exponential backoff / temporary lockout (e.g. lock after N fails in a window), persisted or in-memory with a sweeper. Return uniform 401 and uniform timing regardless of lockout to avoid enumeration.

### [billing-1] Hosting fee 'unsettled' always equals 'accumulated' — settled amount is never subtracted, settle has no effect on the fee view
- **等级**: HIGH  ·  **类别**: ineffective  ·  **置信**: high
- **位置**: `internal/api/admin_handlers.go:353`, `internal/api/me_handlers.go:166`, `internal/billing/fee.go:24`, `internal/billing/settle.go:19`
- **现象/根因**: ComputeHostingFee(consumption, settled, rate) takes a `settled` parameter (tested in fee_test.go, clamps unsettled to >=0), but BOTH production call sites pass settled=0 literally: admin_handlers.go:353 `ComputeHostingFee(consumption, 0, rate)` and me_handlers.go:166 `ComputeHostingFee(consumption, 0, rate)`. Settle() (settle.go) creates a `settlements` row and appends a `billing_ledger` entry, but nothing in the fee path ever reads settled amounts back: there is no query summing settlements/ledger, and `cost_rollup` (the source of `consumption` via SumCostForOwner) is never reduced or reset on settle. Net effect: unsettledUsd is permanently identical to accumulatedUsd (= consumption*rate). After an operator 'settles' a tenant the fee owed never goes down, and re-settling creates duplicate full-gross settlements. This is the core money-correctness defect of the unit.
- **线上证据**: GET /api/dashboard hosting row for tenant 'test': consumptionUsd 211.15934254999988, rate 0.3, feeUsd 63.34780276499996, unsettledUsd 63.34780276499996 — unsettledUsd == feeUsd exactly, confirming settled is always 0 and no settlement is ever deducted.
- **修复建议**: Add a query SumSettledForOwner (SELECT COALESCE(SUM(settled_usd),0) FROM settlements WHERE tenant_id=$1 [AND status='paid']) and pass its result as the `settled` arg to ComputeHostingFee at both call sites; OR have Settle reduce/zero the settled portion of cost_rollup inside the same transaction. Decide on one source of truth (rollup vs settlements) and make settle and the unsettled view consistent.

### [billing-2] Settle ignores periodStart/periodEnd and always settles all-time gross
- **等级**: HIGH  ·  **类别**: logic  ·  **置信**: high
- **位置**: `internal/billing/settle.go:19`, `internal/api/billing_handlers.go:25`, `queries/settle.sql:1`, `internal/dispatch/service.go:524`
- **现象/根因**: settleHandler accepts periodStart/periodEnd and passes them to Settle, which stores them on the settlements row — but the gross is computed by SumCostForOwner: `SELECT COALESCE(SUM(cost_usd),0) FROM cost_rollup WHERE scope_type='owner' AND scope_id=$1` (settle.sql:1), with NO day/period filter. Compounding this, every production rollup write uses Day:"" (logOK service.go:524 and logStream service.go:640 both pass `Day: ""`), so cost_rollup has no real per-day breakdown to filter on even if a filter were added. Result: a settlement nominally for one period actually charges the tenant's entire lifetime accrued cost; repeated period settlements each re-charge the full all-time total. The period fields are decorative.
- **修复建议**: Write a real day string (todayDayStr()) in AddCostRollup instead of "", and add a period-bounded sum query (SUM(cost_usd) WHERE scope_id=$1 AND day BETWEEN $2 AND $3) used by Settle; or, if all-time settle is intended, remove periodStart/periodEnd from the API to avoid implying period scoping.

### [billing-3] settleHandler trusts caller-supplied tenantId with no owner scoping — admin/operator can settle any tenant
- **等级**: HIGH  ·  **类别**: scoping  ·  **置信**: high
- **位置**: `internal/api/billing_handlers.go:13`, `internal/api/router.go:38`, `internal/api/middleware.go:43`
- **现象/根因**: POST /api/admin/settle is gated by requireAdmin, which admits superadmin, admin, AND operator (middleware.go:60). settleHandler reads body.TenantId and calls billing.Settle(... body.TenantId ...) with NO call to scope(r) and no check that the caller owns or may act on that tenant. By contrast the sibling ledgerHandler (billing_handlers.go:39) DOES call scope(r) and forces a non-superadmin to their own tenant. So a scoped admin (e.g. user 'lz', role admin) or an operator can create a 'paid' settlement and a billing_ledger entry for ANY tenant id they pass, including tenants they have no ownership over. This is an authz-gating inconsistency: the read path is scoped, the write path is not. (Write path not exercised live per the read-only rule.)
- **修复建议**: In settleHandler, call owner,all := scope(r); if !all then require body.TenantId == owner (or 403). Alternatively restrict the settle route to requireSuperadmin if only superadmins should settle. Mirror the exact scoping ledgerHandler already uses.

### [auth-session-1] Session role/sub is read from the signed token and never revalidated against DB — 30-day stale privilege after demote/delete/password-change
- **等级**: MEDIUM (原评 high)  ·  **类别**: logic  ·  **置信**: high
- **位置**: `internal/api/middleware.go:24`, `internal/api/middleware.go:60`, `internal/api/middleware.go:75`, `internal/api/auth_handlers.go:43`, `internal/api/auth_handlers.go:13`, `internal/api/user_handlers.go:82`, `internal/db/sqlc/users.sql.go:64`
- **现象/根因**: IssueSession embeds {sub, role, exp} into an HMAC token with a 30-day TTL (sessionTTLSec = 30*24*3600, auth_handlers.go:13,43). Every authz decision (requireSession, requireAdmin role∈{superadmin,admin,operator} middleware.go:60, requireSuperadmin middleware.go:75, scope() middleware.go:43-52) reads p.Role / p.Sub straight from the verified token — there is no per-request DB lookup of the user's current role or existence. There is also no session epoch/version/revocation column on the tenants table and no sessions table. Consequences: (a) setUserRoleHandler demoting a superadmin→tenant (user_handlers.go:82 / SetTenantRole) leaves the victim with full superadmin API access for up to 30 days; a self-promotion isn't possible (requireSuperadmin gates role changes) but a lateral/older token keeps stale power; (b) deleteUserHandler (user_handlers.go:60) removes the DB row but the deleted user's outstanding token still passes requireSession/requireAdmin until expiry; (c) changePasswordHandler rotating the password (user_handlers.go:162) does NOT invalidate other live sessions, so a stolen cookie survives a password reset — the usual 'change password to kick attackers' assumption is false.
- **线上证据**: GET /auth/me with the superadmin cookie returns {"perms":["*"],"role":"superadmin","sub":"u_b95265262b9b7f4815f736ad"} — role/sub served directly from the token; unauth /auth/me returns 401.
- **修复建议**: Add a monotonically-increasing session_epoch (or token_version) to tenants; embed it in the token and compare against the DB value in requireSession, rejecting on mismatch. Bump the epoch on password change, role change, and (implicitly via row deletion) on delete. Alternatively shorten TTL drastically and/or look up the live role per request. At minimum, re-fetch role from DB in requireSuperadmin/requireAdmin so privilege changes take effect immediately.

### [auth-session-3] Session cookie missing Secure flag (and HMAC secret has no strength validation)
- **等级**: MEDIUM  ·  **类别**: security  ·  **置信**: high
- **位置**: `internal/api/auth_handlers.go:44-47`, `internal/config/config.go:38-40`
- **现象/根因**: The tower_session cookie is set with HttpOnly:true and SameSite=Lax but NO Secure flag (auth_handlers.go:44-47). The live instance is served over plain HTTP (http://23.237.28.170:8088), so the 30-day session token already traverses the network in cleartext; absence of Secure also means that if TLS is ever fronted, the cookie would still be sent on any accidental http:// request, enabling interception. ### [fallback-2] Fallback channel MaxConcurrent does not actually cap concurrency — over-limit requests still go through
- **等级**: MEDIUM  ·  **类别**: ineffective  ·  **置信**: high

### [state-store-1] Deleting a node leaves orphan account_state rows that resurrect as ghost accounts (and stale permanent bans) on restart
- **等级**: LOW (原评 medium)  ·  **类别**: data  ·  **置信**: high
- **位置**: `internal/api/admin_handlers.go:163`, `internal/api/admin_handlers.go:169`, `internal/state/store.go:203`, `internal/state/persist.go:29`, `queries/nodes.sql:18`, `queries/account_state.sql:13`, `migrations/20260621000005_account_state.sql:2`
- **现象/根因**: deleteNodeHandler runs q.DeleteNode (queries/nodes.sql:18 = DELETE FROM nodes WHERE id=$1) and Store.RemoveNode (store.go:203, in-memory only). The account_state table (migration 20260621000005) has NO foreign key to nodes and there is NO 'DELETE FROM account_state WHERE node_id=$1' query anywhere. So a deleted node's account_state rows persist forever. On the next restart, LoadStates (persist.go:29) does SELECT * FROM account_state (queries/account_state.sql:13, unfiltered) and calls RestoreBreaker for EVERY row, recreating the deleted node's accounts as ghost entries in s.accts — including any persisted permanent=true ban. Two concrete impacts: (1) account_state grows unbounded over the lifetime of the box (every node ever deleted leaves rows); (2) RemoveNode's promise that deleted accounts 'no longer surface as ghost rows' only holds until the next restart, after which they are back in memory. The dispatch/status UI does hide them (dispatch_status.go:54 drops keys not in keyOwner) and elastic util only sums baselineSet (service.go:393-397), so runtime dispatch is not skewed — this is a persistence-hygiene / unbounded-growth bug, not a routing bug.
- **线上证据**: GET /api/admin/dispatch/status returned only 3 accounts (n_89f3...:two profiles half_open, n_b07d...:one active), all belonging to live nodes; the UI-side keyOwner filter masks any orphans, so the live API cannot reveal whether orphan rows exist in account_state. Static reasoning: no DELETE on account_state exists in queries/ or migrations/.
- **修复建议**: On node delete, also purge persisted state: add 'DELETE FROM account_state WHERE node_id = $1' to queries/account_state.sql and invoke it in deleteNodeHandler (alongside DeleteNode), or add an ON DELETE CASCADE FK from account_state.node_id to nodes(id). Optionally have LoadStates skip rows whose node_id no longer exists in nodes (join/anti-join) so a stale permanent ban can never be resurrected onto a recreated node_id.

### [state-store-2] PersistAll computes the persisted status string from a copy missing CoolUntil and LimitedUntil
- **等级**: LOW  ·  **类别**: data  ·  **置信**: high
- **位置**: `internal/state/store.go:380`, `internal/state/persist.go:13`, `internal/state/account.go:21`
- **现象/根因**: PersistAll (store.go:376-388) builds a shallow Account copy that includes only Breaker, Slots, Disabled, Offline — it does NOT copy CoolUntil or LimitedUntil. SaveState (persist.go:14) then calls a.Status(now), which reads a.CoolUntil (account.go:35) to decide the 'cooldown' status. Because CoolUntil is 0 in the copy, an account currently in a 429 error-cooldown is persisted with status 'active'/'banned' instead of 'cooldown'. So the persisted status row can be momentarily inaccurate but nothing reads it back. Worth noting because the status column looks authoritative but is effectively write-only.
- **修复建议**: Either copy CoolUntil (and LimitedUntil) into the snapshot Account in PersistAll so the persisted status is accurate, or document that account_state.status is advisory-only and never read on warm-start (it isn't).

### [vault-crypto-2] Credential import path stores EMPTY ciphertext — OAuth creds are never persisted (encrypted or otherwise)
